Requerimientos integrales de infraestructura de BI

Fase inicial: tableros y análisis orientados al seguimiento de la cobranza sobre la cartera de las líneas que hoy comercializa La Fe (funerario, vida AP y RCV). Etapas posteriores: se espera ampliar el BI a otras áreas de forma escalonada; la infraestructura debe estar planteada para ese crecimiento. Este documento explica, en lenguaje claro, qué se necesita de tecnología y por qué, sin poner en riesgo el sistema operativo diario.

Qué se solicita y justificación (en breve)

La compañía dispone de un core de seguros (Sirweb) y de su base de datos, donde vive la información operativa diaria. Hoy la oferta se concentra en funerario, vida AP y RCV. En esta primera etapa el área de cobranza necesita indicadores y reportes centrados en el seguimiento de la cobranza (mora, vencimientos, cartera y demás métricas que defina negocio sobre esas líneas) en formato de inteligencia de negocio (BI): gráficos, tablas y filtros que apoyen el día a día, sin depender solo de extractos manuales.

Para lograrlo sin sobrecargar ni poner en riesgo el sistema transaccional, se requiere un entorno separado donde puedan ejecutarse las aplicaciones (pantallas en Streamlit —propuesta de quien elabora este documento para la interfaz—, más una API en Python que concentre las reglas y los números) y donde exista un lugar autorizado para guardar tablas intermedias (resultados de cálculos, agregados) que no deben escribirse sobre la base de solo lectura que suelen dar las réplicas.

La justificación es triple: (1) continuidad operativa — el core sigue atendiendo la operación aseguradora del día a día sin ralentizarse por consultas pesadas de analítica; (2) seguridad y gobierno del dato — accesos acotados, HTTPS, red interna o VPN, y políticas de acceso acordadas (p. ej. SSO en fases posteriores); (3) evolución por fases — el mismo esquema técnico puede incorporar más KPIs y, más adelante, BI en otras áreas de la compañía de forma escalonada, evitando fragmentar la plataforma.

Lo que sigue detalla qué infraestructura concreta se pide (servidor, Docker, bases de datos, red). Al final del documento se incluye un anexo con glosario para quienes no trabajan a diario con estos términos.

1. Objetivo

Disponer de un entorno estable, seguro y escalable para:

  • Tableros y análisis con Streamlit (propuesta de este documento). Cualquier otro front o marco equivalente sería cambio explícito acordado con el área de tecnología. Énfasis inicial en cobranza y posibilidad de ampliar a otras áreas según roadmap.
  • API en Python (p. ej. FastAPI) como capa de métricas y reglas de negocio frente a los datos.
  • No impactar el sistema transaccional del core (Sirweb): la lectura operativa de datos sensibles se hará por canales definidos por tecnología.
  • Infraestructura dimensionada para crecer con el volumen de datos y la concurrencia de acceso, y para incorporar en etapas posteriores nuevos tableros o fuentes alineadas a otras áreas, sin obligar a rediseñar desde cero.

3. Alcance funcional (fase inicial)

  • Contexto de cartera: la oferta comercial actual de La Fe se centra en funerario, vida AP y RCV; los indicadores de cobranza deben alinearse a esas líneas.
  • Área y enfoque inicial: seguimiento de la cobranza (indicadores a definir con negocio: mora, vencimientos, cartera, cumplimiento, etc.).
  • Roadmap: en etapas posteriores se incorporará BI para otras áreas de manera escalada; la infraestructura que se instancie debe contemplar ese crecimiento (capacidad, red, separación de entornos o esquemas según política).
  • Tipo de uso: principalmente lectura y visualización; cualquier escritura hacia sistemas corporativos queda fuera de alcance salvo acuerdo explícito posterior.

5.4 Instalación de dependencias y actualizaciones

El área de tecnología debe definir un único mecanismo, acorde a la política de La Fe, para que el servidor obtenga paquetes Python e imágenes Docker de forma controlada (habitualmente salida vía proxy corporativo gestionado por sistemas). Si no hubiera salida a Internet, se aplicará el procedimiento acordado de importación de artefactos (entorno restringido o importación manual documentada, p. ej. air-gap). No se detallan aquí marcas o productos de repositorio: basta con que exista una vía clara y soportada por la empresa.

5.5 Operación

  • Servicios con arranque automático y reinicio ante fallo.
  • Logs con rotación; preferible integración con centralización de logs si existe.
  • Backups: al menos configuración de despliegue y definiciones versionadas en Git; política de backup de volúmenes Docker acordada con sistemas.

5.6 Monitoreo y alertas

Los logs y su rotación no bastan para enterarse a tiempo si algo falla: se requiere visibilidad proactiva antes de que el fallo se note al abrir el tablero. Se pide disponer de comprobaciones de salud (health checks) sobre al menos la API (Python), el front (Streamlit) y, si aplica, la correcta ejecución de los jobs ETL (p. ej. última corrida, duración anómala o códigos de error).

Cuando un servicio deja de responder o un job crítico no se ejecutó, el entorno debe poder notificar al equipo de desarrollo BI (y a sistemas si la política lo exige). Como mínimo suele bastar correo electrónico hacia una lista de destinatarios definida (p. ej. casillas del equipo y, si aplica, de guardia). La herramienta concreta (monitor, agente, integración) la define tecnología en línea con seguridad y coste.

5.7 Continuidad operativa ante incidentes (RTO y RPO)

La política de backups debe ir acompañada de objetivos negociados con negocio y tecnología sobre cuánto puede estar indisponible el BI y cuánta pérdida de datos es aceptable ante un incidente grave:

  • RTO (Recovery Time Objective, tiempo de recuperación objetivo): tiempo máximo aceptable para volver a tener operativos el tablero y la API tras una caída del servidor u otro evento; importa para la toma de decisiones en cobranza y ayuda a decidir si basta una VM o hace falta alta disponibilidad.
  • RPO (Recovery Point Objective, punto de recuperación objetivo): antigüedad máxima aceptable de los datos recuperados; suele alinearse con la frecuencia de respaldo de la capa analítica y volúmenes.

Los valores concretos de RTO y RPO se fijan con el área de tecnología y se revisan al crecer el uso y la criticidad.

6. Base de datos y datos

6.1 Lectura desde el entorno corporativo

  • Réplica de solo lectura, vistas publicadas por sistemas, o esquema analítico alimentado por ellos.
  • Credenciales dedicadas al proyecto BI, mínimo privilegio (idealmente solo SELECT sobre objetos acordados).
  • Conectividad desde el servidor de apps (o desde el host de ETL) según reglas de firewall.

6.2 Escritura para tablas intermedias

La réplica no admite creación de tablas propias del proyecto. Opción preferida: instancia PostgreSQL (u homólogo aprobado) con escritura en esquema dedicado a BI (idealmente servidor separado del servidor de aplicaciones). Opción alternativa (piloto): DuckDB / SQLite / Parquet en volumen persistente Docker, con acuerdo explícito sobre backup, tamaño y concurrencia.

Ante el volumen histórico potencial de datos de cobranza, el paso desde archivos locales o DuckDB/Parquet a un PostgreSQL dedicado debe basarse en criterios de salida del piloto acordados entre negocio, desarrollo BI y tecnología. A modo orientativo (los umbrales X e Y se definen en el proyecto):

  • Cuando el almacenamiento o la capa analítica supere un tamaño de referencia (p. ej. X GB) o deje de cumplirse el rendimiento acordado.
  • Cuando la concurrencia (accesos simultáneos al tablero y a la API) supere un umbral de referencia (p. ej. más de Y) o haya picos sostenidos que degraden los tiempos de respuesta.
  • Cuando se requieran capacidades que el esquema de archivos locales no cubre bien (p. ej. integridad concurrente, auditoría, respaldos en caliente según política).

Streamlit como capa de presentación: es la opción planteada en este documento para construir el tablero con rapidez en Python. El framework Streamlit no incluye de serie un modelo multiusuario corporativo (cuentas, roles o perfiles de aplicación); implementarlo sería desarrollo adicional sobre el marco. En esta fase el acceso a la URL del tablero se restringe por red corporativa, proxy y políticas acordadas (sección 8), sin asumir ese módulo dentro de la app.

8. Desarrollo, repositorio e integración continua

Usar Git (p. ej. vía GitHub u otra forja aprobada) aporta control de versiones (historial de cambios, ramas, revisiones), trabajo colaborativo y posibilidad de integración y despliegue automatizados. No sustituye el almacén de datos de la empresa: solo concentra el código y la definición técnica del despliegue.

El trabajo diario se realiza en estaciones de desarrollo (p. ej. equipo de escritorio o portátil con capacidad equivalente; ver sección 10). El código se centraliza en un repositorio privado en la forja Git autorizada por la empresa (creado y mantenido por desarrollo BI, con accesos y política acordados con tecnología): API (FastAPI), tableros (Streamlit), contenedores y pruebas.

8.1 Qué sí y qué no debe vivir en el repositorio

En el repo no deben residir datos sensibles de negocio (cartera, pólizas, datos personales), volcados de bases de datos, ni credenciales en claro (contraseñas, tokens, claves API). Esos elementos permanecen en los servidores y bases autorizados o en gestores de secretos (sección 9). El repositorio guarda únicamente lo necesario para construir y ejecutar la aplicación: código, plantillas de configuración sin secretos, Dockerfile, compose y documentación. Así se reduce el riesgo aunque la forja esté en la nube, siempre en línea con la política de seguridad y cumplimiento de La Fe.

8.2 Contenido mínimo esperado en el repositorio

  • Dockerfile (uno por servicio o los que aplique): instrucciones para construir la imagen de la aplicación.
  • docker-compose.yml (u homólogo): orquestación de contenedores en local o integración (app, API, motor analítico del piloto, etc.).
  • .gitignore: reglas para no versionar secretos, credenciales, entornos locales ni artefactos generados.
  • Código fuente (.py), manifiesto de dependencias (requirements.txt o equivalente) y documentación breve de despliegue.

8.3 Despliegue automatizado (p. ej. GitHub Actions)

Una forma eficiente de conectar el repositorio con los servidores Ubuntu es usar GitHub Actions (o pipeline equivalente en la forja elegida) para ejecutar, ante cambios en la rama acordada, verificaciones, construcción de imágenes y despliegue a integración o producción según política de La Fe. Si no se autoriza ejecución desde la nube hacia la red interna, se adoptará la variante aprobada (agente self-hosted, publicación de artefactos y despliegue manual supervisado, etc.).

13. Supuestos y puntos abiertos

  • El modelo exacto de datos de cobranza y nombres de tablas/vistas lo define negocio + tecnología con el core Sirweb, alineado a las líneas funerario, vida AP y RCV.
  • La ampliación del BI a otras áreas y el calendario por fases lo acuerdan negocio y tecnología; la infraestructura descrita debe poder soportar esa evolución sin partir de cero en cada etapa.
  • El dimensionamiento fino puede ajustarse tras un piloto de carga y al incorporar nuevas áreas o mayor uso concurrente.
  • Cualquier cambio que implique escritura en sistemas transaccionales del core queda fuera de este documento hasta nuevo acuerdo.

14. Planificación indicativa (un recurso a tiempo parcial) y fases posteriores

Advertencia: de carácter meramente orientativo. Lo que más condiciona el calendario suele ser infraestructura, datos y aprobaciones. El alcance del proyecto incluye desarrollar las consultas al origen de datos autorizado y los procesos ETL hacia la capa analítica (sección 6).

Supuesto: un recurso a tiempo parcial asignado al proyecto; piloto acotado (cobranza; líneas funerario, vida AP y RCV); y colaboración oportuna del área de tecnología en accesos y despliegue. Un orden de magnitud razonable sería:

  • ~1,5–2,5 meses: primer entregable en integración / pruebas (API + tablero inicial + ETL mínimo).
  • ~3–5 meses: piloto en producción con valor para el área de negocio (despliegue estable, alertas básicas; revisar sección 6.2).
  • Si sistemas o seguridad alargan los trámites, el calendario se desplaza sin relación con la dedicación del recurso.

Definir por escrito un MVP del piloto (pantallas e indicadores) y la fecha objetivo de disponibilidad de datos ayuda a acotar estos plazos.

14.1 Fases posteriores (solo esquema; sin fechas)

  • Fase posterior 1 — Consolidación de cobranza: más KPIs y mayor uso del tablero, monitoreo y backups acordados, salto a motor analítico dedicado cuando proceda (sección 6.2).
  • Fase posterior 2 — Otras áreas de BI: nuevos tableros y fuentes sobre la misma arquitectura, con permisos y gobierno del dato por ámbito.
  • Fase posterior 3 — Madurez: autenticación corporativa si aplica, capacidad y continuidad alineadas a criticidad y RTO/RPO.

14.2 Seguimiento de avances (orientativo)

Conviene acordar una cadencia de seguimiento entre desarrollo BI, tecnología y, cuando aplique, negocio; lo siguiente es meramente orientativo y no sustituye el calendario de reuniones que defina la empresa:

  • Fase de arranque (infraestructura y acceso a datos): sincronización semanal breve (p. ej. 15–30 minutos) o informe escrito de hecho / próximo paso / bloqueos.
  • Revisión con lista de logros o demo corta aproximadamente cada dos semanas, como equilibrio entre visibilidad y carga administrativa.
  • Reunión o informe mensual con dirección o negocio puede reservarse para alcance y prioridades, no para el detalle técnico cotidiano.
  • Si no hay novedades relevantes, la sesión semanal puede cancelarse o sustituirse por comunicación asíncrona.

Ubuntu ServerSistema operativo Linux orientado a servidores. La marca LTS viene de Long-Term Support: versión con soporte prolongado. Versión pedida: 22.04 LTS.
Core / SirwebCore, en jerga de sistemas, es el núcleo o sistema central. Sirweb es el sistema transaccional donde se registra la operación cotidiana. No es el lugar donde conviene ejecutar reportes pesados de BI.
OLTPSiglas de Online Transaction Processing (procesamiento de transacciones en línea): tipo de uso de base de datos pensado para muchas operaciones del día a día (pagos, estados de cuenta). Debe mantenerse ágil.
Réplica (solo lectura)Copia de los datos actualizada para consultar, sin mezclar con la escritura del core (en inglés suele decirse read replica o copia de solo lectura). Ideal para alimentar reportes sin tocar producción.
Base analítica (con escritura)Otro motor de base de datos (o archivos controlados) donde el proyecto BI puede crear tablas propias de resumen; no sustituye al core.
ETLSiglas de Extract, Transform, Load —extraer, transformar y cargar datos—: procesos que extraen datos del origen, los transforman y los cargan donde corresponda (por ejemplo, hacia la base analítica).
HTTPS / TLSHTTPS es HTTP con cifrado (Hypertext Transfer Protocol Secure); TLS es la capa criptográfica (Transport Layer Security, seguridad de la capa de transporte). Navegación cifrada; evita que datos sensibles circulen en claro por la red interna.
Proxy inversoEn inglés reverse proxy: componente delante de las aplicaciones que centraliza el acceso por nombre web, certificados y reglas de entrada.
DNS internoDNS son las siglas de Domain Name System (sistema de nombres de dominio): convierte nombres en direcciones. Aquí se refiere a un nombre estable para el servicio (por ejemplo bi-cobranza.empresa) en lugar de una dirección numérica difícil de recordar.
VPN / intranetVPN viene de Virtual Private Network (red privada virtual). La intranet es la red interna de la organización. Forma de acceder a sistemas internos solo desde redes o equipos autorizados, según política de La Fe.
SSO / directorio (LDAP, Azure AD, etc.)SSO es Single Sign-On (inicio de sesión único). LDAP son las siglas de Lightweight Directory Access Protocol (acceso ligero a directorios). “Azure AD” alude al directorio de Microsoft (hoy a menudo bajo la marca Entra ID). Inicio de sesión unificado con el usuario corporativo, en lugar de contraseñas sueltas por aplicación.
FirewallTérmino inglés que en español se traduce por cortafuegos: reglas de red que permiten solo los caminos necesarios (por ejemplo, de la app hacia las bases de datos autorizadas).
VM (máquina virtual)Siglas de Virtual Machine. “Computadora simulada” dentro del hardware de la empresa; permite asignar CPU, RAM y disco sin comprar un equipo físico nuevo en cada caso.
SSDSiglas de Solid-State Drive (unidad de estado sólido). Disco rápido; recomendable para que consultas y aplicaciones respondan bien.
PostgreSQLBase de datos relacional de uso muy extendido (nombre del proyecto de software libre); candidata habitual para la capa analítica con escritura.
PyPI / registry DockerPyPI es el Python Package Index (índice de paquetes de Python). Un registry de Docker es donde se publican las imágenes de contenedor. En la práctica, el área de tecnología define cómo el servidor accede a ellos (p. ej. proxy o repositorio interno), según la política de la empresa —sin fijar aquí marcas concretas.
RTO / RPORTO (Recovery Time Objective): tiempo máximo aceptable para restaurar el servicio tras una caída. RPO (Recovery Point Objective): cuán “viejos” pueden ser los datos recuperados. Sirven para planificar backups y alta disponibilidad.
Health checkComprobación automática de que un servicio “está vivo” (p. ej. la API responde, el front carga, un job terminó bien). Complementa los logs para detectar fallos antes de que el problema se manifieste en el tablero.
GitHub ActionsServicio de integración continua en GitHub: ejecuta flujos (pruebas, construcción, despliegue) al hacer push o según cron; debe ajustarse a la política de red y seguridad de La Fe.
